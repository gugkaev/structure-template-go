# 🏛 structure-template-go

> Production-ready шаблон Go-проекта на основе принципов **Domain-Driven Design** и **Clean Architecture**.

---

## Философия

Шаблон обеспечивает жёсткое разделение ответственности на четыре слоя. **Домен — это сердце** системы: он ничего не знает ни о HTTP, ни о базах данных, ни о каких-либо внешних системах. Всё остальное адаптируется под него.

```
Внешний мир
      │
  [ Inbound Adapters ]      ← HTTP handlers, gRPC, CLI, consumers
      │
  [ Application / Use Cases ]
      │
  [ Domain ]                ← сущности, value objects, доменные сервисы
      │
  [ Outbound Adapters ]     ← PostgreSQL, Redis, внешние API
      │
Внешние системы
```

Правило зависимостей абсолютно: **внутренние слои никогда не импортируют внешние**.

## Структура проекта

```
.
├── cmd/
│   ├── api/                    # Точка входа HTTP-сервера
│   ├── cli/                    # Точка входа CLI
│   └── migrations/             # Точка входа для запуска миграций
│
├── internal/
│   ├── domain/                 # ← Чистая бизнес-логика. Ноль внешних зависимостей.
│   │   ├── entities/           # Бизнес-объекты с идентичностью
│   │   ├── value_objects/      # Неизменяемые объекты, равенство по значению
│   │   ├── services/           # Stateless доменные сервисы
│   │   ├── errors/             # Доменные ошибки
│   │   └── ports/
│   │       ├── repository/     # Контракты хранилища (интерфейсы)
│   │       └── external/       # Контракты внешних сервисов (интерфейсы)
│   │
│   ├── application/            # ← Оркестрирует домен. Никакого кода фреймворков.
│   │   ├── usecases/           # Один файл — один use case
│   │   └── dto/                # Входные / выходные структуры данных
│   │
│   ├── adapters/               # ← Переводит между внешним миром и приложением.
│   │   ├── inbound/
│   │   │   └── http/
│   │   │       ├── handlers/   # Обработчики маршрутов
│   │   │       ├── middleware/ # Auth, логирование, recovery
│   │   │       ├── dto/        # Структуры HTTP request/response
│   │   │       └── server/     # Настройка сервера и маршрутизации
│   │   └── outbound/
│   │       ├── persistence/    # Реализации репозиториев (БД)
│   │       └── clients/        # Внешние HTTP / gRPC клиенты
│   │
│   └── infrastructure/         # ← Сборка, конфиг фреймворка, DI-контейнер.
│       ├── config/             # Загрузка окружения и конфигурации
│       ├── database/           # Подключение к БД, миграции
│       ├── logger/             # Настройка логгера
│       └── container/          # Внедрение зависимостей
│
├── pkg/                        # Общие утилиты без доменной логики
│   ├── errors/
│   ├── validator/
│   ├── pagination/
│   └── testutils/
│
├── migrations/                 # SQL-файлы миграций
└── docs/                       # API-документация, ADR, диаграммы
```

## Правила слоёв

| Слой | Может импортировать | Не может импортировать |
|---|---|---|
| `domain` | ничего | всё остальное |
| `application` | `domain` | `adapters`, `infrastructure` |
| `adapters` | `application`, `domain` | `infrastructure` |
| `infrastructure` | всё | — |


## Inbound vs Outbound адаптеры

**Inbound** — внешний мир вызывает ваше приложение:
- HTTP/gRPC обработчики
- Консьюмеры очередей сообщений
- CLI-команды

**Outbound** — ваше приложение вызывает внешний мир:
- Реализации репозиториев (БД)
- Клиенты внешних API
- Email, S3, push-уведомления

Обе стороны реализуют **интерфейсы, определённые в `domain/ports/`** — именно поэтому домен остаётся чистым.

## Стратегия тестирования

```
internal/domain/          → unit-тесты, моки не нужны
internal/application/     → unit-тесты, мокировать порты
internal/adapters/        → интеграционные тесты
```

Тесты домена — самые быстрые и самые ценные. Они проверяют бизнес-правила без какой-либо инфраструктуры.
